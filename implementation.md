# Implementation Plan

Companion to [plan.md](plan.md). Nine steps; each has **files**, **what it does**, and a
**verify** action so progress is visible and testable at every point. Steps 1–5 need no
API key and no LLM — everything is deterministic and fast to check.

Module layout (new code lives in one package):

```
src/agent/
  core.py            # existing — gains tools + system prompt wiring (step 7)
  cli.py             # existing — gains tool-call trace (step 7)
  prompts.py         # system prompt playbook (step 7)
  store.py           # step 3: SQLite schema + queries — shared seam, swap point at scale
  embeddings.py      # step 6: embed texts (ingest) / query (search), numpy similarity
  ingest/            # WRITE path (offline). Never imports search/.
    filename.py      # step 1: filename → metadata
    docx_extract.py  # step 2: docx → sections (paragraphs + tables as markdown)
    pipeline.py      # step 3/4/6: build the index; `uv run index`
  search/            # READ path (online). Never imports ingest/.
    tools.py         # step 5/6: the four agent tools (plain functions first)
evals/
  conftest.py  helpers.py  cases/*.yaml  test_retrieval.py  test_agent.py
data/
  corpus/            # the 189 .docx (gitignored)
  index.db           # generated by ingest (gitignored)
  reference/cfr-820-30.md   # curated checklist (step 9, committed)
```

---

## Step 0 — Environment (~10 min)

- Install uv (`brew install uv`), `uv sync`, keys into `.env`.
- Add deps: `python-docx`, `pyyaml`, `numpy`, `rapidfuzz`.
- Add script entry: `index = "agent.ingest.pipeline:main"`.
- **Verify:** `uv run chat` answers a trivial question (baseline agent works).

## Step 1 — Filename parser (~30 min)

(Produces the catalog *fields*; verified standalone before the DB exists. The catalog
itself is persisted as the `documents` table in `data/index.db` at step 3.)

- `filename.py`: `parse_filename(name) -> DocMeta` — doc_id, type_prefix, project,
  number, title, revision, is_signed, is_obsolete, is_duplicate. Handles the known
  suffix mess (`-Obsolete` / `_Obsolete`, `-signed` / `- Signed`, `(1)`, missing rev →
  rev=None treated as current).
- **Verify:** `uv run python -m agent.ingest.filename data/corpus` prints a table of all
  189 parses + summary (docs, revision chains, obsolete count, duplicates) and exits
  nonzero if any filename fails to parse. Eyeball the weird ones (ESF-P01-003, BOM-079).

## Step 2 — Docx extraction (~45 min)

- `docx_extract.py`: `extract_sections(path) -> list[Section]` where Section =
  (heading_path, text, tables_as_markdown). Walk document body in order; split on
  Heading1/Heading2 styles; convert each table to a markdown block; oversized tables
  split by rows with header repeated. Empty doc → returns [].
- **No-heading fallback** (22 real docs incl. both 3P reports): paragraph-window
  chunks with `section_path="body"`.
- **Verify:** `uv run python -m agent.ingest.docx_extract "data/corpus/ECR-577..."`
  dumps sections to stdout. Check: the all-tables ECR yields readable markdown; a
  protocol shows OBJECTIVE / METHODS / ACCEPTANCE CRITERIA paths; empty docs report empty.

## Step 3 — SQLite store + ingest command (~45 min)

- `store.py` schema:
  - `documents(doc_id, revision, title, type, status, path, content_hash, ...)` —
    one row per revision; "latest non-obsolete" is a query, not a column.
  - `chunks(id, doc_id, revision, section_path, position, text)`
  - `chunks_fts` — FTS5 mirror of chunk text (BM25)
  - `refs(from_doc, to_doc, context)` — filled in step 4
- `pipeline.py` (`uv run index`): corpus dir → parse, extract, chunk, dedupe by hash,
  write DB, then print the **summary report**: files → docs/revisions/chains, empty
  flagged, duplicates, chunks, FTS rows.
- Empty docs (24 files, 13%!) get catalog rows with `is_empty=1` and no chunks —
  some docs are empty in EVERY revision (IFU-MX1), so catalog presence is what
  separates "exists but empty" from a wrong "not found."
- Extract software-version strings (`v3.3.0`) from titles into a catalog column —
  enables version-scoped queries for near-zero cost. Title stored per revision
  (titles change across revs).
- **Verify:** the summary report itself, plus a smoke query:
  `sqlite3 data/index.db "SELECT doc_id FROM chunks_fts WHERE chunks_fts MATCH 'dielectric' LIMIT 5"`.

## Step 4 — Reference graph (~20 min)

- In ingest: regex doc-ID pattern (`[A-Z]{2,4}(-P0\d)?-\d+`, plus IFU-MX1-style) over
  chunk text → `refs` edges with the matching sentence as context. Flag dangling
  targets (P00 docs) rather than dropping them.
- **Verify:** `sqlite3` — inbound edges for a risk doc; confirm the VVAM trace matrix
  cites VVPRs; print edge count in ingest summary.

## Step 5 — Tools as plain functions + layer-1 evals bootstrap (~1 hr)

- `tools.py`, no LLM anywhere:
  - `search_catalog(query?, doc_type?, status?)` → ALL matching docs + count; rapidfuzz
    over doc_id and title; groups revision chains; structured signals for
    obsolete-only / no-match cases.
  - `search_content(query, mode="keyword")` → top chunks from FTS5, each with
    (doc_id, revision, section_path, text).
  - `read_document(doc_id, revision=None, section=None)` → default latest non-obsolete;
    returns available revisions + status signals (incl. `is_empty` — "exists but empty,"
    never "not found"). **Size guard:** over ~30k chars returns section outline +
    requested/first section, not the whole doc (trace matrix is 2.9M chars).
  - `get_references(doc_id, direction)` → edges + context sentences.
- Bootstrap `evals/cases/known_item.yaml` + `enumeration.yaml` (~10 cases) and
  `test_retrieval.py` (parametrized over all case files, calls tools directly).
- **Verify:** `uv run pytest evals -k retrieval` — this is the inner loop from here on.
  Also: `search_catalog("bill of materails")` finds BOM-055 (typo test).

## Step 6 — Embeddings + semantic mode (~45 min)

- `embeddings.py`: at ingest, batch-embed header+text per chunk
  (`text-embedding-3-small`), normalize, store BLOBs; at query, embed + dot product
  over the matrix loaded once. Ingest `--no-embeddings` flag keeps everything else
  usable without a key.
- `search_content(..., mode="semantic")` wired to it.
- **Verify:** retrieval eval case: "electrical safety verification acceptance criteria"
  must surface 3P-P01-33 (dielectric/leakage) in top 5 — the vocabulary-mismatch test
  keyword search fails by design.

## Step 7 — Agent wiring + system prompt + CLI trace (~1 hr)

- `prompts.py`: the playbook — QMS vocabulary table, citation format + verbatim-quote
  rule, per-pattern recipes, honesty-over-silent-defaults, ask-vs-assume.
  Vocabulary must warn: MEMO is a junk drawer (SRS, SDS, architecture, Phase Design
  Reviews are all MEMOs; only one doc is type DR) — don't over-trust type filters;
  confirm with title/content search.
- `core.py`: wrap tools with `@tool` docstrings (tool descriptions ARE prompts),
  pass to `create_deep_agent`.
- `cli.py`: stream events; print each tool call live (`→ search_catalog(type='ECR') · 3 results`).
- **Verify:** run demo queries 1 (BOM), 2 (ECR count), 5 (510(k) not-found) by hand in
  `uv run chat`; watch the trace use the right tools; counts must come from the tool.

## Step 8 — Full eval suite (~1–1.5 hr)

- Remaining case files: exploratory, extraction, revisions, cross_document, compliance
  (~4–6 cases each; ground truth from the catalog we now have).
- `test_agent.py` (layer 2): runs the agent per case; checks must_cite / count /
  answer_contains; **citation-validity check on every case** (cited doc exists, quoted
  text appears in it); optional LLM-judge rubric, judge model configurable.
- `helpers.py`: citation parser (regex over `[DOC-ID Rev X, §...]`), metrics collector;
  print a results table at session end (recall@5, count-exact rate, citation validity).
- **Verify:** `uv run pytest evals` → the results table. This table is deliverable #1
  for the 5:00 walkthrough.

## Step 9 — Polish for demo + walkthrough (~1 hr, flexes)

- `data/reference/cfr-820-30.md` + compliance recipe + one gap-detection eval case.
- README quickstart (index → chat → pytest). DESIGN.md from plan.md + eval numbers +
  failure analysis (3–4 known-miss queries and why).
- Demo dry run: the 5-query script end to end. Web frontend only if all above is done.

## Follow-up — ingest unit tests (post-MVP, or if a step-1/2 bug bites)

Distinct from evals: evals measure *search quality*; these are plain unit tests for the
*write path*, in `tests/` (not `evals/`). During the trial, steps 1–3's verify commands
(parse-all-189 report, extraction dumps, ingest summary) cover this ground manually —
these tests make that regression-proof:

- `test_filename.py` — table-driven cases straight from the edge-case list: suffix
  variants (`-Obsolete`/`_Obsolete`, `- Signed`), duplicate `(1)`, missing rev,
  BOM-079-style title versions, typo'd titles parse without error.
- `test_docx_extract.py` — tiny fixture .docx files (all-tables doc, headings doc,
  empty doc): sections split on headings, tables land whole as markdown, empty → [].
- `test_pipeline.py` — mini-corpus of ~6 fixture files → assert summary counts (docs,
  chains, empty skipped, dupes deduped), latest-non-obsolete resolution, graph edges.
